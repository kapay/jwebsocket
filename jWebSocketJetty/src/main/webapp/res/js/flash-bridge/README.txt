              <div class="highlight"><pre><div class='line' id='LC1'>* How to try</div><div class='line' id='LC2'><br/></div><div class='line' id='LC3'>Assuming you have Web server (e.g. Apache) running at http://example.com/ .</div><div class='line' id='LC4'><br/></div><div class='line' id='LC5'>- Download web_socket.rb from:</div><div class='line' id='LC6'>&nbsp;&nbsp;http://github.com/gimite/web-socket-ruby/tree/master</div><div class='line' id='LC7'>- Run sample Web Socket server (echo server) in example.com with: (#1)</div><div class='line' id='LC8'>&nbsp;&nbsp;$ ruby web-socket-ruby/samples/echo_server.rb example.com 10081</div><div class='line' id='LC9'>- If your server already provides socket policy file at port 843, modify the file to allow access to port 10081. Otherwise you can skip this step. See below for details.</div><div class='line' id='LC10'>- Publish the web-socket-js directory with your Web server (e.g. put it in ~/public_html).</div><div class='line' id='LC11'>- Change ws://localhost:10081 to ws://example.com:10081 in sample.html.</div><div class='line' id='LC12'>- Open sample.html in your browser.</div><div class='line' id='LC13'>- After "onopen" is shown, input something, click [Send] and confirm echo back.</div><div class='line' id='LC14'><br/></div><div class='line' id='LC15'>#1: First argument of echo_server.rb means that it accepts Web Socket connection from HTML pages in example.com.</div><div class='line' id='LC16'><br/></div><div class='line' id='LC17'><br/></div><div class='line' id='LC18'>* Troubleshooting</div><div class='line' id='LC19'><br/></div><div class='line' id='LC20'>If it doesn&#39;t work, try these:</div><div class='line' id='LC21'><br/></div><div class='line' id='LC22'>1. Try Chrome and Firefox 3.x.</div><div class='line' id='LC23'>- It doesn&#39;t work on Chrome:</div><div class='line' id='LC24'>-- It&#39;s likely an issue of your code or the server. Debug your code as usual e.g. using console.log.</div><div class='line' id='LC25'>- It works on Chrome but it doesn&#39;t work on Firefox:</div><div class='line' id='LC26'>-- It&#39;s likely an issue of web-socket-js specific configuration (e.g. 3 and 4 below).</div><div class='line' id='LC27'>- It works on both Chrome and Firefox, but it doesn&#39;t work on your browser:</div><div class='line' id='LC28'>-- Check "Supported environment" section below. Your browser may not be supported by web-socket-js.</div><div class='line' id='LC29'><br/></div><div class='line' id='LC30'>2. Add this line before your code:</div><div class='line' id='LC31'>&nbsp;&nbsp;WEB_SOCKET_DEBUG = true;</div><div class='line' id='LC32'>and use Developer Tools (Chrome/Safari) or Firebug (Firefox) to see if console.log outputs any errors.</div><div class='line' id='LC33'><br/></div><div class='line' id='LC34'>3. Make sure you do NOT open your HTML page as local file e.g. file:///.../sample.html. web-socket-js doesn&#39;t work on local file. Open it via Web server e.g. http:///.../sample.html.</div><div class='line' id='LC35'><br/></div><div class='line' id='LC36'>4. If you are NOT using web-socket-ruby as your WebSocket server, you need to place Flash socket policy file on your server. See "Flash socket policy file" section below for details.</div><div class='line' id='LC37'><br/></div><div class='line' id='LC38'>5. Check if sample.html bundled with web-socket-js works.</div><div class='line' id='LC39'><br/></div><div class='line' id='LC40'>6. Make sure the port used for WebSocket (10081 in example above) is not blocked by your server/client&#39;s firewall.</div><div class='line' id='LC41'><br/></div><div class='line' id='LC42'>7. Install debugger version of Flash Player available here to see Flash errors:</div><div class='line' id='LC43'>http://www.adobe.com/support/flashplayer/downloads.html</div><div class='line' id='LC44'><br/></div><div class='line' id='LC45'><br/></div><div class='line' id='LC46'>* Supported environments</div><div class='line' id='LC47'><br/></div><div class='line' id='LC48'>It should work on:</div><div class='line' id='LC49'>- Google Chrome 4 or later (just uses native implementation)</div><div class='line' id='LC50'>- Firefox 3.x, Internet Explorer 8 + Flash Player 9 or later</div><div class='line' id='LC51'><br/></div><div class='line' id='LC52'>It may or may not work on other browsers such as Safari, Opera or IE 6. Patch for these browsers are appreciated, but I will not work on fixing issues specific to these browsers by myself.</div><div class='line' id='LC53'><br/></div><div class='line' id='LC54'><br/></div><div class='line' id='LC55'>* Flash socket policy file</div><div class='line' id='LC56'><br/></div><div class='line' id='LC57'>This implementation uses Flash&#39;s socket, which means that your server must provide Flash socket policy file to declare the server accepts connections from Flash.</div><div class='line' id='LC58'><br/></div><div class='line' id='LC59'>If you use web-socket-ruby available at</div><div class='line' id='LC60'>http://github.com/gimite/web-socket-ruby/tree/master</div><div class='line' id='LC61'>, you don&#39;t need anything special, because web-socket-ruby handles Flash socket policy file request. See</div><div class='line' id='LC64'>http://www.lightsphere.com/dev/articles/flash_socket_policy.html</div><div class='line' id='LC65'>for details and sample script to run socket policy file server. node.js implementation is available here:</div><div class='line' id='LC66'>http://github.com/LearnBoost/Socket.IO-node/blob/master/lib/socket.io/transports/flashsocket.js</div><div class='line' id='LC67'><br/></div><div class='line' id='LC68'>Actually, it&#39;s still better to provide socket policy file at port 843 even if you use web-socket-ruby. Flash always try to connect to port 843 first, so providing the file at port 843 makes startup faster.</div><div class='line' id='LC69'><br/></div><div class='line' id='LC70'><br/></div><div class='line' id='LC71'>* Cookie considerations</div><div class='line' id='LC72'><br/></div><div class='line' id='LC73'>Cookie is sent if Web Socket host is the same as the origin of JavaScript. As such, if you want to support proxied conncetions, you&#39;ll have to supply this information to the WebSocket constructor when Flash is being used. One way to go about it would be to ask the user for proxy settings information if the initial connection fails.</div><div class='line' id='LC85'><br/></div><div class='line' id='LC86'><br/></div><div class='line' id='LC87'>* How to host HTML file and SWF file in different domains</div><div class='line' id='LC88'><br/></div><div class='line' id='LC89'>By default, HTML file and SWF file must be in the same domain.
